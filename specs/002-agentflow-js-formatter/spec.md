# Feature Specification: Agentflow JS 格式化外掛

**Feature Branch**: `002-agentflow-js-formatter`  
**Created**: 2026-03-30  
**Status**: Draft  
**Input**: User description: "Agentflow VS Code 外掛：自動格式化 .js 檔案，支援自訂語法、壓縮多餘空行"

## Clarifications

### Session 2026-03-30

- Q: 含有 `{???:` 自訂語法的檔案中，標準 JS 格式化規則的適用範圍為何？ → A: 最外層（`{???:` 區塊以外）套用完整 JS 格式化；區塊內部保留原始結構，不套用 AST 層級格式化。
- Q: 工作區無任何縮排設定時，外掛應使用哪個縮排單位？ → A: 使用 VS Code 當前作用中編輯器的 `tabSize` 設定（跟隨使用者 / 工作區設定）。
- Q: Agentflow 應如何與 VS Code 格式化提供者機制互動？ → A: 註冊為 `.js` 格式化提供者，不強制設為預設；on-save 僅在使用者將 Agentflow 設為預設格式化工具時觸發。
- Q: 外掛原始碼語言與打包流程為何？ → A: 以 TypeScript 撰寫，先編譯為 JS 再用 `vsce package` 打包 `.vsix`。
- Q: 多層嵌套 `{???:` 區塊的格式化邊界如何定義？ → A: 遞迴邊界——每一層 `{???:` 均為獨立邊界；層降之間的程式碼套用完整格式化；任何層級的 `{KEY:` 開頭行冒號後的換行均不得移除。
- Q: 標準 JS 格式化是否會將 `{ALL:` 重排為 `{\n  ALL:`？ → A: 不允許。`{KEY:` 整行（含開頭的 `{`）為自訂語法單元，格式化器不得對其拆行或重排；格式化器須先識別此模式並整體跳過，再對其餘程式碼套用標準規則。

### Session 2026-03-31

- Q: `{???:` 區塊的結束邊界如何偵測？ → A: 以大括號深度計數（brace depth counting）判斷：遇到 `{KEY:` 後開始計數，當配對的 `}` 使深度歸零時即為區塊結尾。
- Q: 出現在字串或註解內的 `{KEY:` 要視為自訂區塊嗎？ → A: 不視為。僅**獨立成行**（行首或前方只有空白）的 `{KEY:` 才觸發自訂語法識別，字串字面値與註解內的模式一律按標準 JS 規則處理。
- Q: 鍵名允許的字元集為何？ → A: 允許包含空白在內的任意字元，僅排除換行與 `}`；即第一個 `{` 後到 `:` 前的所有非換行非 `}` 字元均為合法鍵名。
- Q: 區塊以外的程式碼包含無效 JS 語法時如何處理？ → A: 放棄整個格式化操作，回傳原始內容不修改，並展示錯誤通知告知使用者。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 儲存時自動格式化 JS 檔案 (Priority: P1)

開發者在 VS Code 中編輯 `.js` 檔案並儲存，外掛自動對整個檔案內容套用格式化規則（縮排、空格、括號標準化、多餘空行壓縮），無需額外操作。

**Why this priority**: 這是外掛的核心價值——讓開發者不需手動執行格式化；儲存時自動觸發是最低摩擦的使用方式。

**Independent Test**: 開啟含有未格式化內容的 `.js` 檔案，儲存後驗證內容是否符合格式化規則，即可獨立測試此功能。

**Acceptance Scenarios**:

1. **Given** 一個包含縮排錯誤的 `.js` 檔案處於開啟狀態，**When** 使用者按下儲存（Ctrl+S），**Then** 檔案內容自動套用標準縮排與空格規則。
2. **Given** 一個包含連續三行以上空白行的 `.js` 檔案，**When** 使用者儲存，**Then** 所有連續空白行皆被壓縮為單一空白行。
3. **Given** 一個 `.ts`（TypeScript）檔案，**When** 使用者儲存，**Then** 外掛不對該檔案執行任何格式化動作。

---

### User Story 2 - 保留自訂 `{鍵名:` 語法不被誤改 (Priority: P1)

開發者的 `.js` 檔案中使用了 `{任意字串:` 後接換行的特殊物件語法（如 `{ALL:\n...}`、`{初始化:\n...}`），格式化後此語法結構應完整保留，鍵名不被加引號、換行不被移除。

**Why this priority**: 此為外掛的核心差異化需求；若格式化程式破壞這個語法，程式碼將無法正確執行，外掛反而造成傷害。

**Independent Test**: 建立包含 `{初始化:\n  doSomething();\n}` 的 `.js` 檔案並觸發格式化，驗證鍵名與換行結構完全未被修改。

**Acceptance Scenarios**:

1. **Given** 一個包含 `{ALL:\n  ...\n}` 語法的 `.js` 檔案，**When** 觸發格式化，**Then** 輸出結果第一行仍為 `{ALL:` 不被拆為 `{\n  ALL:`，鍵名不被加引號，冒號後的換行保持不變。
2. **Given** 一個包含中文鍵名 `{初始化:\n  ...\n}` 的 `.js` 檔案，**When** 觸發格式化，**Then** 中文鍵名保持不變，不視為語法錯誤。
3. **Given** 一個包含混合鍵名（英文、中文、數字）的多個 `{???:` 區塊，**When** 觸發格式化，**Then** 每個區塊的鍵名與結構均完整保留。
4. **Given** 一個合法的普通 JS 物件 `{ key: "value" }`（鍵名後不換行），**When** 觸發格式化，**Then** 該物件仍依標準 JS 規則格式化，不受特殊語法規則影響。

---

### User Story 3 - 打包外掛為可安裝檔案 (Priority: P2)

開發者完成外掛開發後，能夠將外掛封裝為標準 VS Code 外掛安裝檔（`.vsix`），並將該檔案分發給其他使用者透過 VS Code 離線安裝，無需發佈至 Marketplace。

**Why this priority**: `.vsix` 是 VS Code 外掛的標準分發格式；若無法產出可安裝檔案，外掛僅能在開發環境使用，無法交付給最終使用者。

**Independent Test**: 執行打包指令後，確認專案根目錄產出 `.vsix` 檔案，並在另一個 VS Code 實例中透過「從 VSIX 安裝」成功安裝並啟用外掛。

**Acceptance Scenarios**:

1. **Given** 外掛原始碼處於可執行的狀態，**When** 開發者執行打包指令，**Then** 專案目錄下產出一個 `agentflow-<版本>.vsix` 檔案。
2. **Given** 一個 `.vsix` 檔案，**When** 使用者在 VS Code 中選擇「從 VSIX 安裝」並選取該檔案，**Then** 外掛成功安裝並於重載後可正常使用格式化功能。
3. **Given** 打包完成的 `.vsix` 檔案，**When** 檢視其內容，**Then** 包含外掛必要的元資料（名稱、版本、描述）且不含開發環境專用的暫存檔案。

---

### User Story 4 - 手動觸發格式化指令 (Priority: P2)

開發者可透過 VS Code 指令面板（Command Palette）手動觸發 Agentflow 格式化，不依賴儲存動作。

**Why this priority**: 部分開發者偏好手動控制格式化時機，或需要格式化尚未儲存的暫存內容。

**Independent Test**: 於指令面板搜尋並執行「Agentflow: Format Document」，驗證當前 `.js` 檔案內容是否完成格式化。

**Acceptance Scenarios**:

1. **Given** 一個已開啟的 `.js` 檔案，**When** 使用者從指令面板執行「Agentflow: Format Document」，**Then** 檔案內容套用所有格式化規則。
2. **Given** 目前焦點不在 `.js` 檔案（如 `.json` 或其他類型），**When** 使用者執行格式化指令，**Then** 外掛顯示「僅支援 .js 檔案」的提示訊息，並不修改檔案內容。

---

### Edge Cases

- 若 `.js` 檔案為空白或僅含空白行，格式化後應保持空白（或輸出為空），不拋出錯誤。
- 多層巢狀的 `{???:` 區塊：每一層均為獨立邊界，各層之間的程式碼套用完整格式化；任何層級的 `{KEY:` 整行（含 `{`）不得被標準格式化規則重排或拆行；冒號後的換行均不得被移除或合併至下一行。區塊內部包含大括號的小節（如 `if () {}`）不影響深度計數，編譯器以間距括號配對判斷區塊結尾。
- 超大型 `.js` 檔案（數千行）格式化時不應造成 VS Code 明顯卡頓。
- `{???:` 語法假設不會出現在同行行尾；行內物件如 `{ key: value }` 仍依標準規則處理。
- 出現在字串字面値（`"..."` / `'...'` / `` `...` ``）或註解（`//` / `/* */`）內的 `{KEY:` 模式不觸發自訂語法，一律按標準 JS 格式化處理。
- 區塊以外的程式碼包含無法解析的 JS 語法錯誤時，格式化器放棄整例操作并回傳原始內容，同時展示錯誤通知；使用者修復語法後方能再次觸發格式化。

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 外掛 MUST 僅對副檔名為 `.js` 的檔案執行格式化，其他檔案類型不受影響。
- **FR-002**: 外掛 MUST 向 VS Code 註冊為 `.js` 檔案的 `DocumentFormattingEditProvider`；on-save 自動格式化僅在使用者將 Agentflow 設為對應工作區的預設格式化工具時觸發，不強制覆蓋其他工具。
- **FR-003**: 外掛 MUST 提供可從 VS Code 指令面板執行的手動格式化指令（「Agentflow: Format Document」）。
- **FR-004**: 外掛 MUST 對不被任何 `{???:` 區塊包含的程式碼套用完整標準 JavaScript 格式化規則（縮排統一、空格標準化、括號對齊、分號等）。每一層 `{???:` 區塊內部不套用 AST 層級格式化，保留原始結構。
- **FR-005**: 外掛 MUST 將連續兩行以上的空白行壓縮為恰好一行空白行（適用於整個檔案，含 `{???:` 區塊內外）。
- **FR-006**: 外掛 MUST 將符合以下條件的語法識別為自訂區塊起始：該行**獨立成行**（行首或前方只有空白）且符合 `{<鍵名>:` 後接換行的模式，其中鍵名為一個或以上的非換行非 `}` 字元（允許空白、中文、英文、數字等任意字元）。對於出現在字串字面値或註解內的同名模式，一律視為一般 JS 內容不觸發自訂語法。區塊結尾以大括號深度計數判斷（配對的 `}` 使深度歸零）。區塊整體視為不透明單元：不加引號、不拆行、不移除換行、不對內容套用 AST 格式化（適用於所有層級）。
- **FR-007**: 外掛 MUST NOT 將 `{???:` 自訂語法以任何標準 JS 格式化規則重排，包含但不限於：在 `{` 後自動換行、在鍵名前加縮排、對鍵名加引號。
- **FR-008**: 外掛 MUST 在當前焦點非 `.js` 檔案時，手動執行格式化指令後顯示適當的提示訊息。
- **FR-009**: 格式化操作 MUST 具冪等性——相同輸入多次格式化後輸出結果相同。
- **FR-010**: 外掛 MUST 可封裝為 `.vsix` 安裝檔，供使用者在 VS Code 中透過「從 VSIX 安裝」離線安裝，不需發佈至 Marketplace。
- **FR-011**: 当格式化器在 `{???:` 區塊以外的程式碼中遇到無法解析的語法錯誤時， MUST 放棄整個格式化操作，回傳原始文件內容不做任何修改，並透過 VS Code 通知 API 展示錯誤訊息告知使用者。

### Key Entities *(include if feature involves data)*

- **格式化規則集（FormattingRuleSet）**：定義縮排、空格、括號等標準 JS 格式規則，以及自訂語法保留規則。
- **自訂語法模式（CustomSyntaxPattern）**：識別獨立成行的 `{<鍵名>:\n` 的模式，鍵名為一個或以上的非換行非 `}` 字元（含空白、中文、英文等），用於在格式化流程中豁免或特殊處理。
- **文件格式化器（DocumentFormatter）**：接收 `.js` 文件內容、套用規則集、回傳格式化後內容。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一般 `.js` 檔案（≤ 1000 行）儲存後格式化完成時間不超過 500 毫秒，不影響儲存操作的流暢感。
- **SC-002**: 含有 `{???:` 自訂語法的檔案，經格式化後 100% 的自訂鍵名與結構保持不變（零破壞率）。
- **SC-003**: 格式化操作具備冪等性——同一檔案連續格式化兩次，第二次結果與第一次完全相同。
- **SC-004**: 外掛在 VS Code 閒置時不造成可感知的效能影響——`activate()` 中不啟動任何 `setInterval`/`setTimeout` polling；所有事件監聽透過 VS Code `context.subscriptions` 訂閱機制管理；閑置時不執行任何持續週期性程式碼。
- **SC-005**: 開發者能在不閱讀任何說明文件的情況下，於 5 分鐘內完成安裝並成功觸發第一次格式化。
- **SC-006**: 打包指令執行後能在 60 秒內產出可用的 `.vsix` 檔案，且該檔案可直接在乾淨的 VS Code 環境中安裝並正常運作。

## Assumptions

- 格式化縮排單位完全依照 VS Code 當前作用中編輯器的 `tabSize` 設定（`editor.tabSize`）；外掛不自行預設縮排值，亦不讀取 `.editorconfig`。
- `{???:` 自訂語法不會出現在同一行的行尾（即冒號後必定換行）；行內物件如 `{ key: value }` 仍依標準規則處理。
- 外掛目前範圍僅支援本地端 `.js` 檔案格式化；遠端工作區（Remote SSH / WSL）不在 v1 範圍內。
- 外掛以 **TypeScript** 撰寫，編譯後產出 JS 再由 `vsce package` 打包為 `.vsix`；`vsce package` 執行時會自動觸發 `npm run compile`。
- 目標 VS Code 版本為 **1.75 或以上**（2023 年初後的穩定版本）。
- 外掛不主動處理與其他格式化工具（如 Prettier、ESLint autofix）的衝突；外掛註冊為格式化提供者而非強制預設，使用者需自行在 VS Code 設定中指定 Agentflow 為預設格式化工具方能觸發 on-save。
- 术語說明：本文件中 `{???:`、`{KEY:`、`{鍵名:` 均指相同的自訂語法結構，正式符號記法為 `{Key: }`（鍵名區塊）；三者可互換使用。
